Uusi Aurinko 模组开发日志（第一周）

发表于 2021-05-23 更新于 2021-05-24 阅读次数：

前言

事实证明，在完全熟练之前，遵循业界中的共识的能极大帮助一个新手。就拿模组开发举例，如果在开始制作之前先草拟一份企划案，既可以方便小组内的各成员交流想法，也可以方便自己在制作时查阅自己先前的想法。这样的一份企划案不用太细致，只要能传达清楚自己的点子，就能起到很大的帮助作用。

出于同样的原因，我决定每周在我的博客撰写一篇开发日志（devlog）。目的有二：一是把开发过程中零散的思路、遇到的困难和对应的解决方案记录下来，方便自己或其他人以后参考；二是对外公布阶段性的成果，接受评论和建议并且给自己继续做下去的动力。

概况

模组的名字已经被选定为 Uusi Aurinko。小组的名字是 Ilmarinen（伊尔玛利宁），取自于芬兰神话中能锻造与发明任何东西的不死者。目前小组里有三个人：我（程序），XeKr（美工），Neubulaeko（吉祥物）。

在第一周里，模组的企划案 已经完成。参赛队伍与作品也在 TeaCon 网站 上完成了注册（可以在这里 看到）。代码仓库 已经公开发布到了 GitHub 上面。大部分模组物品已经拥有了贴图并加入了游戏，并且其中两个物品的代码已经大致完成。

美术设计

目前所有元素石头和太阳种子、太阳石头的贴图已经由 XeKr 制作完成。这些贴图在参考 Noita原作贴图的基础上做了一些改进。原作中的水石、火石、太阳种子和太阳石的轮廓都是完全一致的；而且太阳石和地石上画有符号，而其它石头没有。模组中的贴图则做了统一化和差异化，移除了太阳种子、太阳石和地石上面的符号，并且重置了外形。

以掉落物实体存在的元素石头会拥有粒子特效。例如火石会有火焰粒子，水石会有滴水粒子。

邪眼和微型月球的贴图已经完成，但还没有加入游戏中。贴图都会比较类似原作中的形象。微型月球还会额外拥有一个正方体的实体建模。

太阳的贴图和模型还没有开始动工，但是初步确定为一个正方体的，会缓慢自转的实体。

物品设计理念

企划案中有少量关于物品设计理念的叙述，这里详细地说一下。

模组中的物品虽然有着诸多特殊效果，但这些效果大多是为了增加难度，让模组游玩流程更加难以控制。除了基本只有正面效果的水石以外，物品都会或多或少地增加麻烦：火石会烧掉你的建筑物、掉落物和生物；电石会伤害饲养的动物并且可以在水中杀死没有电石的玩家；地石会将方块变成泥土并有可能造成方块坍塌；粪石使持有者中毒，反胃，还将液体转化为难以利用的粪水。太阳种子和太阳石则更加混沌：直接将随处可见的方块引爆或替换成火焰。微型月球也可能将怪物快速吸到身边造成麻烦。

程序设计

鼠标悬浮在模组中绝大部分的物品上面时，除了会显示黄色高亮的物品名以外，还会有“更多信息：按下 <Shift>”的字样。当玩家按下 Shift 按键时，则会显示详细的物品描述。这个设计借鉴自 tetra 模组，可以防止玩家移动物品时被较长的物品描述干扰。

模组中的大部分物品既能够在玩家的手中生效，又能够以掉落物的形式在世界中生效。这一类物品在代码中被称为 ItemRadiative，即“放射性物品”。为了方便玩家在养成太阳的时候将物品远距离投入太阳，这些物品都有右键远距离投掷功能。它们的代码框架基本搭建完毕：在手中或世界产生特效，在世界中产生粒子效果，以及右键投掷。

火石和水石在游戏中的效果基本制作完成。火石的效果与企划案中的设计相差无几，只是进行了一些细化，提高了可燃方块的着火概率并能直接点燃附近实体。水石则根据游戏性和代码实现难度做出了一些额外的修改。水石将岩浆源和流动的岩浆分别变成黑曜石和圆石的设计可能导致玩家将地狱的岩浆海完全铺上黑曜石，因此被削弱为将岩浆源变成类似岩浆块 + 霜冰的半凝固岩浆方块。半凝固岩浆会随时间变回岩浆，并且在地狱中这一过程会被大大加速。“身上非信标效果双倍速度消退”的设计也因为代码难以较好实现而放弃。

在参考原版霜冰编写半凝固岩浆方块代码的时候，我发现了维基百科上这行字所描述的特性的实际作用。

当霜冰融化或被摧毁时，任何相邻的霜冰，在它相邻的6个面剩余的霜冰少于2个时就会立即融化。

假如没有这个设计，或将“立即融化”变更为 age 加 1,都会导致残留下大量消融极慢且孤立的霜冰方块。虽然立即融化显得比较突兀，但是却能解决这一问题。

电石只完成了很小一部分。它的闪电传播特效初步确定为在一定范围内相邻的导电方块与空气接触的表面上生成闪电的粒子效果。导电方块的连续判定采用了从电石坐标开始的广度优先搜索，耗时不是很大。闪电粒子效果还没有绘制，麻痹和闪电免疫效果也没有制作。

放弃药水加速消退效果

水石原来有“身上非信标效果双倍速度消退”的设计，但是由于原版药水效果的底层代码原因作罢。周期性生效的效果，诸如生命恢复、中毒等，是通过对剩余时间模运算（\(\%\)）来判断是否需要生效一次的。因此如果让它以双倍速度消退，仅仅会加快药水的生效速度。

如果略微修改原版代码，让药水每 tick 最多生效一次，依然有以下 bug：当生效间隔与每 tick 消退速度的最小公倍数小于两者之积的时候，生效次数依然不符合预期。举个例子：当剩余时间最初为 \(200\),生效间隔为 \(50\) 的时候,如果每 tick 剩余时间减少 \(1\),有 \(200, 150, 100, 50, 0\) 四个时间点可能触发效果（实际上 \(200, 0\) 这两个时间点不会触发）。如果每 tick 减少 \(2\)，由于 \(50\) 和 \(2\) 的最小公倍数也是 \(50\)，依然有 \(200, 150, 100, 50, 0\)四个可能的触发时间点。

最后的挣扎是将药水的剩余时间粗暴减半，然而这也有问题。玩家可以通过反复拿出水石将药水剩余时间多次减半。只是短暂地拿出放下水石就能减半药水效果，平衡性也会受到影响。

综上所述，唯一的方案是增加一个单独的“下次生效倒计时”标签然后利用 Mixin 重写原版药水生效机制的全部代码。这样会极大地修改原版代码，改变药水效果的底层行为。导致其它一些可能依赖原版底层行为的模组、数据包出错。因此考虑过这些代价之后，我选择暂时放弃这部分功能的实现。以后如果与多余时间或者更好的方法也许会重新实现这个功能。

反射，Access Transformer 和 Mixin

当需要调用或修改原版 Minecraft 代码中一些 private 方法、参数时，常见的方法有三个：反射，Access Transformer 和 Mixin。在询问了 TeaCon 群的 3Tusk，CrispyChips 等前辈，再加上自己的理解，基本上总结出了这三种方法的利弊。

反射

反射是 Java 本身自带的一种技术。它允许程序员通过获取并操作类、方法和变量的对象进行修改 private 变量，调用 private 方法，甚至将 final 变量进行修改。

但是这里并不推荐修改声明为 final 的变量。详细原因可以在 这里 看到，我这里简述一二。对于一个变量来说，final 关键字不仅代表无法修改，还涉及由此延伸的一系列与 JVM 有关的特性。如果一个变量无法修改，那么 JVM 可以直接进行缓存，无视可能的修改。即使 final 变量被修改，JVM 中的缓存可能不会变化，导致代码无法产生预期的效果。

另外，在使用反射时应当对第一次反射操作获取到的变量、方法对象进行缓存，并在后续的操作中尽可能直接使用缓存的对象。获取这些反射对象的性能消耗较大，且可能产生一些错误（因此还需要有额外耗时的 try-catch 语句），进行缓存可以提升性能。

编写 Minecraft mod 时还会涉及到源码的混淆：开发时使用的是反混淆的变量、方法名，但一般使用时使用的是未反混淆的名称。反射时还应该使用 Forge 提供的 ObfuscationReflectionHelper 类中的方法，来避免造成不必要的麻烦。

Access Transformer 与 Mixin

Access Transformer（简称AT）与 Mixin 都是通过两种直接修改 Minecraft 字节码或代码，制作 Coremod ，达成操作 private 方法与变量的效果。由于被这两种方法修改的类在被 ClassLoader 加载前就已经变化，可以避免反射修改 final 变量时的一些问题。这两种方法的实际运行效率相差不多并都快于反射。但是我个人并不推荐 AT，原因主要有三。

首先，虽然 AT 是 Forge 自带的一种修改方法，配置起来比 Mixin 方便许多，但是每次编辑配置文件后都需要重新反编译、反混淆 MC 源码，实际使用的时候反而会比 Mixin 麻烦。

其次，由于 AT 是通过直接修改反编译后的代码来使编译器将私有的变量、方法看作公有的变量、方法，程序员可能会难以直接看出哪些操作涉及了 AT，程序员可能会将修改过的原版类当作未修改的。而 Mixin 是通过注解、Mixin 类进行修改的，因此在使用被修改的类时看到形如 ((MinecraftClassMixin) MinecraftClass).modifiedMethod() 的调用形式，不会污染 Minecraft 源码。

另外，Mixin 还可以注入、重定向、重写原版的方法，实现比 AT 更多样且强大的功能。

结论

不频繁的操作使用 ObfuscationReflectionHelper 进行简便反射操作；频繁的操作使用 Mixin 进行复杂但运行速度快的操作。

原版的火焰 bug

在编写火石代码的时候我查阅了较多的原版火焰方块的代码，并且发现一处 bug。这个 bug 已经被提交为 MC-225779 具体效果是火焰在潮湿地形中燃烧速度反而更快。具体原因与 Forge 开发关系不大，这里就不说了。有兴趣的可以自行查看。

后续计划

下个星期开始，由于学习原因我没有之前那么多的时间用于模组开发，开发难度会放缓。预计五月六月初能完成模组物品的基本代码。在这之后会先专注进行太阳实体的制作。如果有剩余时间会制作“遗忘者”boss。

其他不成熟想法

这里附上一些没有仔细考虑和讨论的不成熟想法。

自旋不为 1 的石板：这块石板的正面和反面都写有文字。当玩家阅读了反面的文字，想转回正面时，会发现“正面”已经换上了新的内容。这个设计为模组的引导手册加入了一些科幻的设定（也可以说是魔幻，全看怎么解释）。更重要的是，它通过设定在某种程度上合理化了一个手册上百页的夸张情况。这个想法的具体实现是做出一个旋转石板，然后原先“正面”的文字渐变成新的文字的效果。

装饰性方块：由于 TeaCon 需要开发者搭建场馆并展示，可以在模组中额外添加一些装饰性（也许也可以有功能性）的方块。
